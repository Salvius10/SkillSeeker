# Product

## Register

product

## Users

DevOps and engineering employees (~200 people) at one company. Two roles: **employee** (browses challenges, picks and submits work, tracks points and rank) and **admin** (creates challenges, reviews submissions, analyzes program health). Employees are engineers: they're comfortable with technical concepts, CLI tools, and code — but they come to this platform to participate in a lightweight internal program, not to configure software.

## Product Purpose

Skill Seeker is an internal challenge platform that motivates engineering skill growth through friendly competition. Employees earn points by completing technical challenges posted by admins. The program runs continuously; success looks like high participation rates, timely admin reviews, and employees who return week after week to pick up new work.

## Brand Personality

Technical · Focused · Rewarding

The platform should feel like the best-designed internal tool the team has ever used — not corporate, not childish. It carries the spirit of a well-run developer hackathon: there's genuine craft in the UI, the point system feels like real recognition, and the experience doesn't waste engineers' time.

## Anti-references

- **Corporate HR portals** (SAP SuccessFactors, Workday): sterile gray enterprise grids, form-heavy flows, no personality. Skill Seeker must feel nothing like these.

## Design Principles

1. **The tool disappears into the task.** Employees come to find a challenge and submit their work. Every screen should make that two-step flow feel effortless; chrome and decoration that slow it down are wrong.
2. **Earned credibility.** This is a technical audience. Pixel-perfect details, tight type, and a coherent system signal craft. Sloppy details signal the platform isn't serious.
3. **Points feel like recognition, not gamification.** The orange point values and leaderboard carry real motivational weight. Over-gamifying (confetti, streaks, badges everywhere) cheapens it.
4. **Admins are experts, not babysitters.** The admin surface should be dense and efficient. Review workflows, analytics, and challenge creation should favor speed over hand-holding.
5. **Urgency is real.** The urgent/deadline mechanic is meaningful; it should read as genuine priority, not a marketing trick.

## Accessibility & Inclusion

WCAG AA: 4.5:1 contrast for body text, keyboard navigable primary flows, screen reader labels on interactive elements, and `prefers-reduced-motion` respect for all animations. This is internal tooling — it should set the accessibility standard the team would expect from any external product.
